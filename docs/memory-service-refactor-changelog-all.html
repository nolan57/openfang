1. System Overview

The Memory Service system consists of three core packages, providing a three-layer memory architecture for flexible long-term and short-term memory management.

3
Memory Layers
10+
Database Tables
30+
Learning Modules
8
LLM Call Points

Session Memory

Session-level temporary memory with TTL expiration, SQLite persistence, multi-keyword search, and vector semantic retrieval.

Evolution Memory

Long-term memory from self-evolution process, using vector embeddings for semantic search and similarity matching.

Project Memory

Project-level knowledge and patterns, using knowledge graph for entity relationships and AST for code entity parsing.

Key Finding

LLM is primarily used for cognitive tasks (extraction, generation, evaluation, detection), while vector embedding storage uses simple bag-of-words model (simpleEmbedding, 384-dim).

9. Summary

Round 1

Implemented basic SQLite persistence, search enhancement, error handling enhancement.

Round 2

Improved cross-platform compatibility, migration resumability, code type safety.

Round 3

Introduced AST code analysis, non-blocking migration, vector dimension guard.

Key Improvement Metrics

Improvement Before After
Code Parsing Regex AST (@babel/parser)
Migration Blocking sync Background async + checkpoint
Vector Dimension Hardcoded 384 Configurable + validation
Error Handling Implicit empty return Explicit NamedError
Session Persistence In-memory Map SQLite persistence

Breaking Changes

  • SessionMemoryService constructor no longer accepts in-memory Map
  • Missing parameters now throw errors (instead of silently returning empty)
  • Unknown memory types throw errors
  • Session memory now requires metadata.sessionId
